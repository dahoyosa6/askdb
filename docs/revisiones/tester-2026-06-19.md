## 3. Calidad de los tests

**Lo bueno (mayoría):**
- Aserciones **estrictas**, no de humo. Ej.: F3 verifica el contenido exacto del history y que el mensaje de fallo NO contiene `error_pg`, `does not exist`, `Traceback` ni `SELECT`. Eso prueba la regla dura de "no fugar info".
- Tests de seguridad **negativos y positivos** (bloquea Y no genera falsos positivos). Raro y valioso.
- Buen uso de espías para verificar *que algo NO se llamó* (rama texto pura no toca chart/spreadsheet; cliente real no se crea si se inyecta).
- Deterministas: `now` inyectable en RateLimiter, nombres de archivo por hash, sin azar.
- Los mocks están a nivel de módulo y **no ocultan bugs**: el de F1 (PK/FK vía pg_catalog) se atrapó precisamente porque los tests pasaron a aserciones estrictas (estaba como `xfail` enmascarando el bug del arquitecto — bien corregido).

**Lo mejorable:**
- `test_telegram_bot.py` declara que "la lógica de los handlers está factorizada en funciones puras". Es **parcialmente cierto**: la orquestación (abrir archivo, elegir send_photo/document/message, el try/except que sanea el error al usuario) vive en el handler async y **no tiene cobertura**. Las funciones puras prueban las decisiones, no el pegamento.
- `test_main_webhook.py` parchea `Update.de_json` para devolver `object()`, así que **no se prueba que un update real se enrute al handler correcto**; solo que `process_update` se llama. Es correcto para el unit del webhook, pero deja un tramo sin cubrir entre el POST y el handler.
- Ningún test frágil detectado. No hay tests que "pasen sin probar nada".

## 4. Lo NO testeado: handlers async (riesgo y recomendación)

**Decisión del proyecto:** sin `pytest-asyncio`; la lógica se factorizó a funciones puras síncronas y esas sí se prueban.

**Evaluación del riesgo:** la mitigación es **buena pero no completa**. Lo que SÍ queda cubierto por funciones puras: allowlist, rate limit, decidir_envio, el puente síncrono, los helpers de voz. Lo que queda **sin red de seguridad** dentro de `handle_text`/`handle_voice`:

- El **orden de los guards** (allowlist → rate limit → vacío) — si alguien reordena y mete el rate limit antes de la allowlist, un no autorizado consumiría cupo. No hay test.
- El **try/except frontera** que garantiza que cualquier excepción → `logger.exception` + `_MENSAJE_GENERICO` (regla dura "nunca fugar el error"). Es el punto más sensible del bot y **no tiene test**. Una regresión aquí (p. ej. dejar escapar la excepción, o mandar `str(exc)` al usuario) no la atraparía la suite.
- El **manejo de archivos** (`open(plan.file_path)` para photo/document): si `file_path` es None o el archivo no existe, hoy revienta y cae al except genérico — comportamiento correcto pero no verificado.
- En voz: el camino "transcripción vacía → `_TEXTO_VOZ_NO_ENTENDIDA`" y "voz demasiado larga → rechazo" antes de gastar Groq.

**Recomendación: SÍ, añadir `pytest-asyncio` antes de producción** (esfuerzo bajo, ~half-día). Es una dependencia de dev estándar, no toca el runtime. Mínimo viable: 6–8 tests con `AsyncMock` del `context.bot` que cubran, por handler:
1. no autorizado → se envía `_TEXTO_NO_AUTORIZADO` y NO se llama al cerebro.
2. rate-limited → `_TEXTO_RATE_LIMIT` y NO se llama al cerebro.
3. camino feliz texto → `send_message` con el texto del router (y **nunca** con SQL).
4. el cerebro lanza excepción → se envía exactamente `_MENSAJE_GENERICO` y se logueó (verificar que el `str` de la excepción NO viaja al usuario).
5. voz: duración > máx → `_TEXTO_VOZ_MUY_LARGA`; transcripción vacía → `_TEXTO_VOZ_NO_ENTENDIDA`.

Razón de fondo: el bot es el **único punto de contacto con el usuario** y la regla dura de "no fugar SQL/errores" se cumple precisamente en ese except no probado. Vale la red de seguridad antes de exponerlo en vivo.

## 5. Guardrails — casos adversariales que faltan (verificados en vivo)

Probé contra el validador real vectores que **los 41 tests actuales NO cubren**. Resultado (capa app):

| Vector | Capa app (`validate_and_secure`) | Capa DB (rol read-only) |
|--------|----------------------------------|--------------------------|
| `SELECT pg_read_file('/etc/passwd')` | **PASA** | **BLOQUEA** (`permission denied for function pg_read_file`) |
| `SELECT lo_import('/etc/hosts')` | **PASA** | **BLOQUEA** (`permission denied for function lo_import`) |
| `SELECT * FROM pg_shadow` (hashes de credenciales) | **PASA** | **BLOQUEA** (`permission denied for view pg_shadow`) |
| `SELECT * FROM pg_user` | **PASA** | permitido (datos no sensibles) |
| `SELECT current_setting('data_directory')` | **PASA** | permitido (info de infra; menor) |
| `SELECT * FROM dblink('...','...')` | **PASA** | **BLOQUEA** salvo que exista la extensión + privilegio |
| `TABLE orders`, `VALUES (1),(2)`, `(SELECT 1)` | BLOQUEA (no empieza por SELECT/WITH) | — |

**Lectura del hallazgo (importante, matizado):**
- **NO es un agujero de escritura.** La garantía read-only de F2 se mantiene: nada de esto modifica datos.
- **El daño real (leer archivos del servidor, robar hashes) lo corta el rol DB**, no la app. La defensa-en-profundidad **funciona**: confirmado en vivo que las tres funciones peligrosas rebotan con `InsufficientPrivilege`. Este es el comportamiento esperado y es buena noticia.
- **Pero la capa de app deja pasar estos vectores en silencio**, igual que pasa `pg_sleep` (ya documentado). Si en el futuro el proyecto corriera con un rol con más privilegios (o se conectara a una DB de cliente mal configurada), la app no sería la barrera. Hoy el único muro es el rol.

**Recomendación (no bloqueante para F8, pero antes de vender a un cliente):** añadir tests que documenten este comportamiento (como ya se hizo con `pg_sleep`) y, opcionalmente, una **denylist de funciones peligrosas por nombre** en la capa app (`pg_read_file`, `pg_read_binary_file`, `lo_import`, `lo_export`, `dblink`, `copy`, `pg_ls_dir`). Esto convierte la app en una segunda barrera real y deja constancia escrita de la decisión. Casos de prueba a agregar: los 6 de la tabla de arriba.

- [ ] **Texto — dato simple:** "¿cuántos pedidos hay?" → responde un número en lenguaje natural (esperado ~830). Sin SQL visible.
- [ ] **Texto — gráfica:** "top 5 clientes por facturación" → llega una **imagen PNG** (barras) con caption legible.
- [ ] **Texto — serie temporal:** "ventas por mes de 1997" → gráfica de **línea**.

- [ ] **Voz:** enviar una **nota de voz** con una pregunta → el bot ecoa `🎤 Entendí: "..."` y luego responde con el formato correcto.
- [ ] **Voz demasiado larga:** nota de voz > `MAX_VOICE_DURATION_S` (120s) → mensaje "la nota es muy larga", NO procesa.
- [ ] **No autorizado:** desde un chat que NO está en `ALLOWED_CHAT_IDS` → responde "No estás autorizado", y el cerebro NO se ejecuta (revisar logs: no debe haber llamada a Anthropic/DB).

- [ ] **/reset + follow-up con memoria:** preguntar "¿cuántos pedidos hay en 1997?", luego "¿y en 1998?" → el segundo entiende el contexto sin repetir. Después `/reset` → "olvidé el contexto" → repetir "¿y en 1998?" → ya NO debe entender el follow-up (memoria limpia). Confirma que la memoria funciona Y que /reset la borra.
- [ ] **Comandos:** `/start` y `/help` responden los textos correctos (con ejemplos).
- [ ] **Pregunta imposible / sin datos:** "¿cuál es el color favorito de mis clientes?" → mensaje saneado en español, **nunca** un stacktrace ni SQL.
- [ ] **Seguridad webhook:** un POST a `/webhook` sin el header secreto → 403 (probar con curl). El `GET /` health → 200.
- [ ] **Errores en prod:** dejar Sentry conectado (opcional) y revisar que no aparezcan excepciones no controladas tras la ronda de pruebas.
